前端 i18n 自动化方案演进：弃用脚手架，转向 Skill Helper 和本地 CLI

上一篇《前端 i18n 自动化实践》已经把人工处理文案的步骤收敛成了一套项目内自动化方案：由脚本负责提取中文、生成英文、生成 key、产出 TMS Excel，再配合 pnpm lang 和替换脚本完成源码替换。

它解决了“能不能自动化”，但还没有很好地解决“怎么低成本复用”和“怎么稳定地在多个项目里使用”。

所以基于上个版本做了一些优化。

方案一：完全使用 AI 完成

但实际用下来问题也比较明显。

首先是慢。完全让 AI 做扫描、判断、翻译、生成 key 和 Excel，整体耗时经常在二十多分钟或更久，而且不太稳定。

其次是 Excel 格式不可靠。TMS 导入 Excel 不只是看表头，很多时候还会受到 sheet 名、表头行、列顺序、格式等影响。AI 从零生成的 Excel 看起来字段没问题，但实际导入 TMS 可能失败。

还有一个问题是结果不稳定。AI 每次生成的英文翻译、待处理文案判断，可能会有细微差异。对于一次性处理少量文案可以接受，但如果大量文案需要处理，还是不够稳定。

各方案优缺点总结

方案 适合场景 优点 缺点
项目内脚手架 单项目深度集成 流程完整，命令清晰 多项目复用困难，污染项目
完全 AI 快速验证想法 启动最快，不写工具 慢，不稳定，Excel 容易出问题
AI + Excel 模板 少量文案处理 Excel 导入稳定 仍依赖 AI，耗时不稳定
Skill Helper 个人 Codex 工作流 不污染项目，本机可复用 依赖 AI 和 Codex
本地 CLI 个人日常稳定使用 全自动，不依赖 AI，输出稳定 只在本机可用，需要维护
独立 Git + pnpm link 工具开发调试 调试方便，不依赖 npm 平台 团队版本不统一
独立 Git + Git 依赖 团队使用方案 版本可控，团队可安装 依赖 Git 权限和 tag 管理
内部 npm 包 长期方案 最规范，使用体验最好 需要 npm 私服

最终选择

目前我保留两套方案。

1、Skill Helper：

适合在 Codex 里处理一些临时场景，用于临时任务、探索场景、需要 AI 判断的场景。

2、本地 CLI：

作为主流程，用于稳定处理当前分支新增中文，自动调用阿里云翻译，并生成 TMS 可导入 Excel。

如果后续团队也要接入，可以把本地 CLI 抽成独立 Git 仓库，先用 pnpm link 调试，再通过 Git 依赖安装给团队使用。

这次实践下来，最大的感受是：AI 很适合探索流程，也适合做一些不确定性的判断；但高频、固定、可复用的流程，还是应该沉淀成工具。

i18n 这种任务尤其明显。扫描、去重、缓存、额度、Excel 输出，都应该交给程序。翻译可以接机器翻译，最后再人工 review。
