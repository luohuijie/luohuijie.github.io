前言

为什么使用 Tailwind CSS

在实际前端开发中，我们经常会遇到一些样式维护上的问题：

  • 使用前端框架后，页面样式一般会使用组件隔离，但组件之间会存在很多重复样式，导致类似样式被反复编写
  • 类名命名是一件很头疼的事。英文命名不一定准确，拼音命名可读性差，名字太长有点啰嗦，取得太短又不知道是什么东西。尤其大多是写业务，很多场景本身就很难抽象出清晰语义，最后就会变成“类名看不懂，样式也不好维护”
  • 团队维护的全局样式很多都没有被充分使用，有点浪费，也容易出现每个人写法不统一的问题
  • 如果 UI 设计规范不够统一的话，经常会出现各种不同的间距、宽高、颜色等，如果每个样式都单独取名，维护成本很高，但直接写内联样式，又不够优雅

而这些问题，Tailwind CSS 都能在一定程度上帮助我们解决。

目前团队很多项目都有使用 Tailwind CSS，用起来也很方便，而且 Tailwind CSS 4 已经很稳定了，所以想把项目升级一下，看看有哪些变化，以及如何升级，升级后有哪些好处

Tailwind CSS 4 升级了什么

Tailwind CSS 4 不是简单的版本升级，而是把安装方式、构建方式、配置方式和主题管理都做了调整。

所以这篇文章主要会根据几个问题展开：

  • v4 到底解决了什么问题
  • 升级后有哪些好处
  • 老项目应该怎么迁移
  • 哪些场景不适合直接升级

一、为什么考虑升级到 Tailwind CSS 4

在 Tailwind CSS 3 中，项目变大之后，样式维护主要会遇到这几个问题：

  • tailwind.config.js 越来越大
  • 样式相关的配置不是集中在一个地方，而是分散在多个文件里
  • 颜色、字体、间距、断点等设计变量分散在 JS 和 CSS 中，维护不够统一
  • 主题管理不够灵活，如果需要修改主题，需要修改 JS 配置，同时需要重新构建样式
  • 样式构建变重，项目变大后，需要扫描更多文件、处理更多配置和样式，构建速度可能会变慢

Tailwind CSS 4 就是在解决这类问题。
官方文档中提到，v4 重点包括：

  • 更高性能的引擎
  • 更统一的工具链
  • CSS-first configuration
  • 自动内容检测
  • 更现代的 CSS 能力支持

所以这次升级不只是“类名升级”，更像是一次样式工具链升级。

二、升级到 v4 有哪些好处

1. 构建和热更新更快

v4 最大的变化是性能提升。项目启动、样式构建、开发时热更新都会更快，特别是在改 class 之后，样式反馈会更快，开发等待时间更少。

2. Vite 项目接入更简单

如果项目使用 Vite，v4 的接入方式会比 v3 更顺，可以直接通过 @tailwindcss/vite 插件接入，不再把主要流程放在 PostCSS 链路里。这样样式构建路径更短，也更容易理解。

3. 配置更统一

v4 更推荐 CSS-first configuration，很多主题配置都可以写到 CSS 的 @theme 中管理。这样配置不再过度依赖 tailwind.config.js，也更接近 CSS 本身。

4. 老项目可以渐进迁移

不需要一次性重写所有配置，老项目可以先通过 @config 继续复用原来的 tailwind.config.js，等稳定后，再逐步把主题配置迁移到 @theme。

三、升级之前，先确认浏览器兼容性

这一步非常重要。

Tailwind CSS v4 的官方浏览器要求是：

  • Chrome 111+
  • Safari 16.4+
  • Firefox 128+

如果项目还需要兼容更老的浏览器，建议继续留在 v3.4。(tailwindcss.com)

所以项目是否升级应该先看真实的用户环境，不能只看技术栈本身。

六、升级中容易踩的坑

1. 项目能跑，不代表升级完成

很多问题不是构建失败，而是页面样式细节变了。
前面提到的页面回归很重要，尤其要注意这些默认行为变化：

  • ring
  • outline
  • border
  • shadow
  • space-x
  • space-y

2. Vue 单文件组件里的 @apply 需要注意

如果继续在 Vue 单文件组件的 <style> 中使用 @apply，建议显式使用 @reference：

因为组件 <style> 会被单独处理，想访问主样式文件里的主题变量和 utilities，需要显式 reference。(tailwindcss.com)

3. SCSS 可以保留，但不建议深度绑定 Tailwind

Tailwind v4 官方并不推荐和 Sass、Less、Stylus 做深度叠加。

但如果项目里只是保留普通 SCSS，例如局部样式、复杂选择器、第三方组件覆盖，仍然可以共存。需要谨慎的是在 SCSS 中大量写 @apply、@theme、theme() 等 Tailwind 相关逻辑。

七、适不适合升级

适合升级的项目

  • 主要面向现代浏览器
  • 本身使用 Vite
  • Tailwind 主要写在模板 class 中
  • 想让样式链路更轻、更统一

建议谨慎升级的项目

  • 还要兼容旧浏览器或旧 WebView
  • 大量依赖 SCSS + Tailwind 混写
  • 组件 <style> 里大量 @apply
  • 历史样式包袱较重

结语

Tailwind CSS v4 在升级配置时可能会遇到一些小波折，但一旦理解了 @theme 和 @config 的分工与协作，就会发现 v4 带来了更灵活、更强大的配置体验。

不过对于老项目，更推荐使用渐进迁移的方式。
